# 문장군 콘텐츠 운영 원칙 v1.0

> 목적: 문장군 리뷰릴스엔진, 블로그엔진, 캐러셀엔진이 많은 콘텐츠를 만들어도 흔들리지 않도록 하는 공통 콘텐츠 철학을 정의한다.
> 최종 업데이트: 2026-06-19

## 1. 최상위 목적

문장군 콘텐츠의 목적은 중문을 설명하는 것이 아니다.

문장군을 홍보하는 것도 아니고, 제품 스펙을 소개하는 것도 아니다.

문장군 콘텐츠의 목적은 고객이 자신의 문제를 발견하게 만드는 것이다.

사람은 중문에 먼저 관심을 갖지 않는다. 사람은 소음, 반려동물, 아이 안전, 냄새, 외풍, 후회 없는 선택, 추가 비용에 관심을 갖는다.

문장군은 중문을 판매하는 회사지만, 콘텐츠에서는 중문을 먼저 판매하지 않는다. 고객의 문제를 보여주고, 그 문제가 왜 생겼는지 설명하고, 해결 과정을 보여준다.

한 문장으로 요약하면 아래와 같다.

```text
문장군 콘텐츠는 중문을 설명하는 콘텐츠가 아니라, 고객의 문제와 사건을 보여주는 콘텐츠로 전환한다.
```

## 2. 핵심 원칙

### 원칙 1: 정보보다 사건을 우선한다

잘못된 접근:

- 중문 종류 설명
- 중문 장점 설명
- 중문 가격 설명
- 회사의 시공 역량 자랑

올바른 접근:

- 고양이가 현관으로 나가려는 상황
- 아이가 중문 턱에서 불편을 겪는 상황
- 입주 후 중문을 미룬 것을 후회하는 상황
- 시공을 포기할 뻔한 현장 구조
- 견적이 달라져 당황하는 상황

사람은 정보를 소비하기 전에 사건에 반응한다. 정보는 사건 뒤에 붙을 때 힘이 생긴다.

### 원칙 2: 후킹은 정보가 아니라 자기 연관성이다

콘텐츠를 본 고객이 "그래서?"라고 생각하면 실패다.

좋은 콘텐츠는 고객 이야기를 보여주되, 보는 사람이 "내 이야기일 수도 있겠다"라고 느끼게 만든다.

따라서 후킹은 제품명보다 상황에서 시작한다.

약한 시작:

- 3연동 중문 설치 후기
- 중문 종류 3가지
- ABS도어 장점

강한 시작:

- 고양이가 자꾸 현관으로 나가려는 집
- 24평 아파트 중문 비용이 달라지는 이유
- 문틀만 바꾸려다 견적 기준이 바뀌는 이유

### 원칙 3: 문제를 먼저 보여주고 해결은 나중에 보여준다

사람은 해결책보다 문제에 먼저 반응한다.

문장군 콘텐츠는 아래 순서를 우선한다.

```text
문제 -> 공감 -> 긴장 -> 해결
```

아래 순서는 피한다.

```text
제품 -> 설명 -> 장점
```

제품은 문제를 설명한 뒤 등장해야 한다. 문장군의 강점도 고객 문제가 드러난 뒤에 나와야 설득력이 생긴다.

## 3. 문장군 콘텐츠 공식

모든 엔진은 아래 구조를 기본값으로 본다.

```text
사건
-> 공감
-> 반전
-> 해결
-> 결과
```

조회수가 나오는 콘텐츠에는 아래 요소가 있다.

- 사건이 있다.
- 문제가 있다.
- 궁금증이 있다.
- 결과가 있다.

조회수가 약한 콘텐츠는 대체로 아래에 머문다.

- 제품 설명
- 시공 설명
- 회사 자랑
- 기능 설명
- 가격 설명

가격과 기능을 다루더라도 사건과 문제 안에 넣어야 한다.

## 4. 절대 금지 방향

아래 표현을 콘텐츠의 중심에 두지 않는다.

- 우리는 최고입니다.
- 우리는 전문입니다.
- 우리는 잘합니다.
- 우리는 15,000건 했습니다.

문장군의 리뷰 수, 시공팀, 무료 실측, 브랜드스토어, 시공 속도는 중요한 신뢰 요소다. 다만 이것들은 주인공이 아니다.

고객은 항상 아래 질문을 먼저 한다.

```text
그래서 내 문제와 무슨 상관인데?
```

브랜드 강점은 고객의 문제를 해결하는 근거로만 사용한다.

## 5. 고객이 실제로 궁금한 것

문장군 콘텐츠는 아래 질문 중 하나 이상을 해결해야 한다.

- 내가 후회할까?
- 내가 손해 볼까?
- 우리 집도 가능할까?
- 반려동물은 안전할까?
- 아이 동선에 불편하지 않을까?
- 소음은 줄어들까?
- 냄새나 외풍은 달라질까?
- 왜 다른 집은 중문을 설치할까?
- 견적이 왜 집마다 다를까?
- 지금 사는 집에서도 시공이 가능할까?

이 질문과 연결되지 않는 제품 설명은 우선순위를 낮춘다.

## 6. 콘텐츠 품질 평가 기준

각 항목은 0~5점으로 평가한다. 총점 25점 중 20점 미만 콘텐츠는 발행하지 않는다.

| 항목 | 질문 | 기준 |
| --- | --- | --- |
| 사건성 | 사건이 존재하는가 | 상황, 갈등, 불편, 선택의 순간이 있어야 한다 |
| 자기 연관성 | 고객이 자신의 이야기로 느끼는가 | 평수, 생활 상황, 가족, 반려동물, 비용 걱정 등과 연결한다 |
| 궁금증 | 다음 내용을 보고 싶어지는가 | 이유, 반전, 놓친 기준이 있어야 한다 |
| 공감 | 실제 경험처럼 느껴지는가 | 과장보다 생활 언어와 구체적 상황이 중요하다 |
| 해결 | 명확한 해결이 존재하는가 | 체크 기준, 선택 기준, 다음 행동이 있어야 한다 |

운영 기준:

```text
20점 미만: 발행 보류
20~22점: 보강 후 발행
23~25점: 발행 후보
```

## 7. 리뷰릴스 총괄 운영 원칙

리뷰는 증거가 아니라 사건 데이터다.

리뷰를 읽을 때는 "좋은 말이 있나"가 아니라 "왜 이 고객이 리뷰를 썼나"를 찾는다.

찾아야 하는 것:

- 갈등
- 불편
- 후회
- 고민
- 놀람
- 감동

좋은 소재:

```text
고양이가 자꾸 현관으로 나갔어요.
```

약한 소재:

```text
3연동 중문 설치했습니다.
```

릴스 기본 공식:

```text
사건
-> 문제 확대
-> 반전
-> 해결
-> 고객 반응
```

릴스 후킹은 제품명이 아니라 사건으로 시작한다.

약한 후킹:

```text
3연동 중문 설치 후기
```

강한 후킹:

```text
고양이 한 마리가 이 집을 바꿨습니다.
```

리뷰릴스 KPI:

- 시청 유지율
- 완주율
- 저장률
- 공유율

조회수는 결과 지표다. 조회수만 보지 않는다.

## 8. 블로그 총괄 운영 원칙

블로그의 목적은 검색 노출만이 아니다. 검색으로 들어온 사람을 설득하는 것이다.

블로그 기본 공식:

```text
문제 제기
-> 공감
-> 사례 또는 상황
-> 해결 기준
-> 결론
```

블로그에서 제품명과 가격 키워드는 필요하다. 다만 제목과 도입부는 검색자의 문제와 연결되어야 한다.

예시:

- 평수별 중문 비용은 평수보다 현관 구조가 먼저입니다.
- 문교체비용은 문짝 하나 가격으로 끝나지 않습니다.
- PVC 걸레받이는 방수성보다 마감 연결을 먼저 봐야 합니다.

블로그는 근거 없는 실제 사례, 직접 인용, 수치 결과를 쓰지 않는다. 사례를 쓰려면 발행 제어 폴더의 `EVIDENCE.json`과 연결한다.

## 9. 캐러셀 총괄 운영 원칙

캐러셀의 목적은 짧은 시간 안에 문제를 인식시키고 저장할 이유를 만드는 것이다.

캐러셀 기본 공식:

```text
1장: 문제
2장: 문제 확대
3~5장: 원인
6~8장: 해결
마지막: 행동 유도
```

캐러셀은 설명 카드가 아니라 판단 카드여야 한다.

약한 방향:

- 중문 종류 설명
- 제품별 장점 나열
- 회사 장점 나열

강한 방향:

- 우리 집이 3연동보다 스윙이 맞을 수 있는 이유
- 중문 견적이 달라지는 구조 변수
- 걸레받이 교체 전 놓치기 쉬운 마감 연결

## 10. 문장군 콘텐츠의 경쟁력

문장군 콘텐츠의 경쟁력은 중문 지식만이 아니다.

진짜 경쟁력은 현장 데이터다.

문장군이 가진 자산:

- 실측 데이터
- 고객 후기
- 시공 사례
- 실패 사례
- 고민 사례
- 견적 전환 과정
- 현장 구조 변수

경쟁사는 제품 설명을 할 수 있다. 문장군은 실제 고객의 문제와 현장 변수를 이야기할 수 있다.

단, 공개 콘텐츠에서는 개인정보, 원본 상담, 원본 리뷰, 관리자 통계, 고객 주소, 고객 실명, 원본 사진을 직접 노출하지 않는다. 공개 저장소에는 익명화된 요약과 불투명한 근거 ID만 남긴다.

### 블로그의 현장 데이터 기본값

블로그 글은 통계 검색어만 따라가는 정보글로 끝내지 않는다. 검색어는 입구이고, 설득은 실제 현장 이야기에서 나온다.

따라서 신규 블로그 글은 아래 질문을 먼저 통과해야 한다.

| 질문 | 기준 |
| --- | --- |
| 이 검색어 뒤의 실제 고객 문제는 무엇인가 | 비용, 구조, 소음, 동선, 시야 차단, 교체 범위처럼 생활 문제로 번역한다 |
| AppSheet에서 찾을 수 있는 현장 조건은 무엇인가 | 제품, 평수, 지역, 구조 변수, 사진 라벨을 정한다 |
| 사진으로 보여줄 수 있는 장면이 있는가 | Before/After, 신발장 간섭, 유리 질감, 문틀 손상, 마감 경계 등 |
| 실제 사례처럼 쓸 수 있는 근거가 있는가 | `EVIDENCE.json`의 불투명 `evidence_ref`와 익명화 범위가 있어야 한다 |

현장 근거가 있으면 `문제 -> 현장 상황 -> 판단 기준 -> 해결`로 쓴다.

현장 근거가 없으면 실제 고객 사례처럼 꾸미지 않는다. 이때는 `현장에서 자주 확인되는 조건 -> 독자가 확인할 체크포인트`로 낮춰 쓴다.

098번 이후 블로그 신규 글은 예전 019/050번대 글처럼 현장 상담을 재현하는 방식으로 쓴다. 독자가 "우리 집도 저럴 수 있겠다"라고 느끼도록 현관 폭, 신발장, 스위치, 문틀, 타일 경계, 유리 질감 같은 구조 변수를 본문 안에 넣는다.

길이는 공백 제외 1,500~2,500자를 합격 범위로 본다. 다만 길이 자체가 콘텐츠의 목적은 아니다. 글 방향은 고객 문제, 현장 변수, 문장군 실제 취급 범위가 먼저이며, 분량 맞추기 때문에 억지 설명을 늘리지 않는다.

사진은 본문에 지시문으로 넣지 않는다. 본문은 네이버에 바로 붙여넣을 글만 남기고, 필요한 컷과 AppSheet 후매칭 방향은 문장 구조 안에 자연스럽게 녹인다.

019번대처럼 제목 후보, 검색 의도, 품질 채점, 예상 성능, 이미지 지시가 한 파일에 같이 있던 방식은 과거 운영 방식이다. 지금은 발행 사고를 줄이기 위해 `posts/`에는 최종 본문만 남긴다. 다만 제작노트로 파일을 쪼개지도 않는다. 문장군 블로그의 기준은 `posts/NNN_키워드.md` 단일 발행 MD이며, 사진 큐와 내부 메모는 남기지 않는다.

## 11. 앞으로의 방향

줄인다:

- 스펙 중심 콘텐츠
- 제품 중심 콘텐츠
- 설명형 콘텐츠
- 회사 자랑형 콘텐츠

늘린다:

- 사건 중심 콘텐츠
- 고객 중심 콘텐츠
- 스토리형 콘텐츠
- 문제 발견형 콘텐츠
- 선택 기준형 콘텐츠

최종 목표:

```text
중문에 관심 없는 사람도 끝까지 보는 콘텐츠를 만든다.
그 결과로 문장군을 신뢰하게 만든다.
그 결과로 문의가 발생한다.
```
