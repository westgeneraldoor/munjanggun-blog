# 블로그 통계자료 자동 수집 워크플로우

> 목적: 사용자가 `블로그 통계자료 수집해`라고 요청했을 때, 작업자가 네이버 블로그 관리자 통계창에 직접 접속해 필요한 통계를 자동으로 취득하되 매번 과한 전체 수집으로 운영이 무거워지지 않게 한다.
> 기본 원칙: 매번 많이 보는 것이 아니라, 매번 충분히 보고, 필요할 때만 깊게 본다.
> 보안 원칙: 원본 XLSX/CSV/XLS, 관리자 화면 캡처, 고객 원본, AppSheet 원본은 공개 저장소에 커밋하지 않는다.

## 1. 트리거 계약

사용자가 아래처럼 요청하면 이 문서가 자동 발동한다.

```text
블로그 통계자료 수집해
```

이 요청의 의미는 다음과 같다.

```text
로그인된 Chrome 세션으로 네이버 블로그 관리자 통계에 접속한다.
기본은 L2 표준 daily로 전일 확정값 중심의 핵심 지표만 수집한다.
이상징후, 주간/월간 판단, 다운로드 필요성이 있으면 L3/L4로 자동 승격한다.
원본은 로컬 전용으로 두고, 프로젝트에는 비식별 요약과 운영 판단만 남긴다.
```

기본 진입 URL:

```text
https://admin.blog.naver.com/doorgeneral/stat/today
```

로그인 세션은 사용자가 미리 살려둔다. 로그인 만료, CAPTCHA, 본인확인, 권한 확인이 뜨면 작업자는 우회하지 않고 사용자에게 처리 요청 후 이어간다.

## 2. 수집 레벨

| 레벨 | 이름 | 시간 예산 | 다운로드 | 언제 쓰나 |
| --- | --- | ---: | --- | --- |
| L0 | 접속 점검 | 1~3분 | 없음 | 로그인/권한/통계 접근 가능 여부만 확인할 때 |
| L1 | 빠른 관찰 | 5분 이내 | 없음 | 이미 daily가 있고 신규 발행글 반응만 볼 때 |
| L2 | 표준 daily | 10~15분 | 없음 | 사용자가 `블로그 통계자료 수집해`라고만 말했을 때 기본값 |
| L3 | 주간/월간 다운로드 | 20~30분 | 허용 | 주 1회, 월초/월말, 7/14/30일 추세 판단 때 |
| L4 | 긴급 심층진단 | 사전 고지 후 수행 | 허용 | 급락/급등, 보호글 이탈, 통계 모순, 전략 재진단 때 |

L3/L4의 지표 다운로드는 사용자의 사전 승인 범위 안에 있다. 별도 재확인 없이 진행할 수 있으나, 진행 업데이트에서 승격 사유와 다운로드 수행 사실을 짧게 알린다.

## 3. 자동 선택 기준

### L0 접속 점검

아래 상황이면 통계 수집 전에 L0만 수행한다.

- 로그인 세션이 살아 있는지 불확실하다.
- 네이버 관리자 페이지 접근 여부만 확인하면 된다.
- 로그인, CAPTCHA, 본인확인, 권한 문제가 보인다.
- 브라우저 제어가 가능한지 확인하는 단계다.

산출물은 짧은 상태 보고만 한다. daily report는 만들지 않는다.

### L1 빠른 관찰

아래 조건이면 L1로 충분하다.

- 최신 daily report가 이미 있다.
- 전일 확정값을 새로 기록할 필요가 없다.
- 신규 발행글의 당일 실시간 반응만 빠르게 본다.
- 사용자가 "방금 올린 글 반응만 봐줘"처럼 좁게 요청했다.

확인 범위:

- 일간 현황의 실시간 조회수, 공감, 댓글, 이웃증감
- 주요 유입경로 상위 일부
- 검색어 상위 일부
- 게시글 조회수 TOP20 중 신규 글 위치

L1은 다운로드하지 않는다.

### L2 표준 daily

`블로그 통계자료 수집해`의 기본 실행값이다.

확인 범위:

| 묶음 | 확인 항목 |
| --- | --- |
| 일간 현황 | 조회수, 동영상 재생수, 공감수, 댓글수, 이웃증감수 |
| 방문 분석 | 조회수, 순방문자수, 방문 횟수, 평균 사용 시간, 재방문율 |
| 유입분석 | 전체, 검색 유입, 사이트 유입 상위 |
| 게시글 반응 | 조회수 TOP20, 공감수 순위 상위, 댓글수 순위 상위 |
| 시간대 분석 | 대표 시간대 1~2개의 유입경로/검색어 |
| 사용자 분석 | 기기 비중, 필요 시 성별·연령 핵심 비중 |
| 신규 글 관찰 | 3일 TOP40, 7일 잔존, 공감/댓글 공명 신호 |

L2 원칙:

- 전일 확정값을 중심으로 기록한다.
- 당일 실시간값은 별도 표기하고 전일 확정값과 섞어 결론 내리지 않는다.
- 공감/댓글은 SEO 성공 확정이 아니라 문제상황 공명 신호로만 쓴다.
- 시간대별 유입어는 전체를 매일 훑지 않고 대표 시간대만 본다.
- L2는 다운로드하지 않는다.

### L3 주간/월간 다운로드

아래 조건 중 하나라도 해당하면 L3로 승격한다.

- 마지막 지표 다운로드가 7일을 넘었다.
- 주간 진단일이다.
- 월초 또는 월말이다.
- 7일, 14일, 30일 추세를 판단해야 한다.
- 순방문자수, 평균 사용 시간, 재방문율 흐름이 필요하다.
- channel diagnosis나 장기 목표 분석 근거가 필요하다.

다운로드 대상:

- 조회수
- 순방문자수
- 방문 횟수
- 평균 방문 횟수
- 재방문율
- 평균 사용 시간
- 유입분석
- 시간대 분석
- 성별·연령별 분포
- 기기별 분포
- 이웃 방문 현황
- 이웃 증감수
- 이웃 증감 분석
- 국가별 분포
- 조회수 순위
- 공감수 순위
- 댓글수 순위

동영상 지표는 현재 재생/시청 데이터가 거의 없으므로 기본 다운로드 대상에서 제외한다. 동영상 데이터가 의미 있는 수준으로 발생하면 L3 대상에 다시 포함할 수 있다.

### L4 긴급 심층진단

아래 조건 중 하나라도 해당하면 L4로 승격한다.

- 일 조회수가 최근 기준 대비 20% 이상 급락하거나 급등했다.
- 검색 유입 비중이 10%p 이상 하락했다.
- Q-001, Q-002, Q-005, Q-006 핵심 보호/공격축 글이 TOP20에서 이탈했다.
- 신규 글 공감은 강한데 3일/7일 조회 잔존이 없다.
- 원본 통계, daily report, 관리자 화면 수치가 서로 모순된다.
- 사용자가 `전략 재진단`, `이상징후`, `전체 통계 다시 봐`라고 요청한다.
- 2,000뷰 목표 갭이나 채널 병목을 다시 판단해야 한다.

확인 범위:

- 관리자 통계 좌측 메뉴 전체
- 각 메뉴의 내부 탭
- 일간, 주간, 월간 비교
- 유입경로 급변
- 검색어 급변
- TOP20 이탈/진입
- 공감/댓글 반응과 조회수 잔존의 불일치
- 필요 시 L3 다운로드 대상 전체

L4는 사전 고지 후 수행한다. 단, 사용자가 이미 다운로드를 승인했으므로 다운로드 자체를 다시 묻지는 않는다.

## 4. 기준일 표기

보고서 상단에는 반드시 기준일을 나눠 적는다.

```text
보고일: YYYY-MM-DD
전일 확정 데이터 기준일: YYYY-MM-DD
당일 실시간 확인 시각: YYYY-MM-DD HH:mm
주간/월간 기준 기간: YYYY-MM-DD~YYYY-MM-DD
수집 레벨: L2 표준 daily
```

전일 확정값과 당일 실시간값을 섞어 결론 내리지 않는다. 당일 실시간값은 "오늘 오전/오후 현재 흐름"으로만 사용한다.

## 5. 저장 규칙

원본 저장 위치:

```text
data/naver/daily/YYYY-MM-DD/
data/naver/raw/
```

공개 저장소에 남길 수 있는 위치:

```text
outputs/reports/daily/YYYY-MM-DD_seo_watch.md
outputs/reports/channel_diagnosis_YYYY-MM-DD.md
outputs/reports/topic_candidates/YYYY-MM-DD_topic_scorecard.md
outputs/reports/REPORT_INDEX.md
outputs/reports/LATEST_CHANNEL_DIAGNOSIS.md
```

원본 XLSX, CSV, XLS, PNG, JPG, WEBP, 관리자 화면 캡처, 고객 원본, AppSheet 원본은 공개 저장소에 커밋하지 않는다. 공개 저장소에는 비식별 요약 리포트와 운영 판단만 남긴다.

다운로드 후 확인 명령:

```powershell
git ls-files "data/naver/daily/**" "data/naver/raw/**" "*.xlsx" "*.xls" "*.csv"
git status --ignored --short data/naver
```

첫 번째 명령 결과가 비어 있어야 한다.

## 6. 산출물 기준

| 상황 | 산출물 |
| --- | --- |
| L0 | 짧은 접속 상태 보고 |
| L1 | 짧은 관찰 보고. 필요 시 daily에 추기 |
| L2 | `outputs/reports/daily/YYYY-MM-DD_seo_watch.md` |
| L3 | daily + 필요 시 weekly/monthly 요약 또는 channel diagnosis |
| L4 | channel diagnosis 또는 별도 심층 진단 보고서 |

daily report는 증거 로그다. 전략 판단은 topic scorecard와 `ACTIVE_TOPIC_QUEUE` Q-ID로 닫는다.

## 7. 판단 원칙

| 지표 | 쓰임 |
| --- | --- |
| 조회수 | 목표 갭 확인 |
| 순방문자수 | 실제 도달 규모 확인 |
| 평균 사용 시간 | 검색 의도와 글 품질 일치도 확인 |
| 방문 횟수/재방문율 | 허브와 내부링크 회수 여부 확인 |
| 검색 유입 비중 | SEO 채널 체력 확인 |
| 시간대별 유입어 | 고객 불안 문장 발굴 |
| 성별·연령·기기 분포 | 제목 톤과 본문 구조 설계 |
| 공감/댓글 순위 | 문제상황 공명 신호 |
| ranking report | weekly/experimental 보조 |

제외 키워드는 유입이 있어도 글감으로 승격하지 않는다. 현관문/방화문, 중문파티션, 무타공 셀프, 자재판매 오해가 있는 소재는 `POSTING_EXCLUSION_RULES.md`를 우선한다.

## 8. 실패 처리

| 상황 | 처리 |
| --- | --- |
| 로그인 만료 | 사용자에게 로그인 요청 후 이어간다 |
| CAPTCHA/본인확인 | 사용자에게 처리 요청. 우회하지 않는다 |
| 네이버 UI 변경 | 가능한 핵심 지표만 수집하고 누락 항목을 보고한다 |
| 다운로드 실패 | 화면 기반 요약으로 degrade하고 실패 사유를 기록한다 |
| 일부 탭 데이터 없음 | 데이터 없음 자체를 기록하고 의사결정에서 제외한다 |
| 동영상 데이터 임계치 미만 | SEO 판단에서 제외한다 |
| 원본 파일 Git 추적 발견 | 즉시 커밋 대상에서 제외하고 보안 정책을 따른다 |

## 9. 마무리 체크

통계 수집 후 아래를 확인한다.

- 수집 레벨과 기준일을 보고서에 적었는가
- 전일 확정값과 당일 실시간값을 분리했는가
- 원본 다운로드 파일을 공개 저장소에 남기지 않았는가
- daily의 다음 액션이 Q-ID로 닫혔는가
- 중복/카니발은 POSTING_REGISTRY 기준으로 확인했는가
- 제외 키워드는 승격하지 않았는가
- ranking을 단독 근거로 쓰지 않았는가
- 필요 시 `npm run ops:daily` 또는 `npm run ops:weekly`를 실행했는가
