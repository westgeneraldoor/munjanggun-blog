# 문장군 블로그 콘텐츠 작업 플레이북

> 목적: 어떤 세션에서 시작해도 같은 기준으로 신규 글 작성, URL 등록, 순위 점검, 리라이팅 판단을 이어가기 위한 운영 문서.
> 최종 업데이트: 2026-06-24

## 1. 세션 시작 체크

새 세션에서 블로그 작업을 시작하면 먼저 아래 파일을 순서대로 읽는다.

1. `AGENTS.md`
2. `docs/strategy/_context.md`
3. `docs/strategy/CONTENT_PLAN.md`
4. `docs/strategy/POSTING_REGISTRY.md`
5. `docs/strategy/CONTENT_OPERATING_PRINCIPLES.md`
6. `docs/operations/APPSHEET_FIELD_STORY_WORKFLOW.md`
7. `docs/strategy/POSTING_EXCLUSION_RULES.md`
8. `docs/operations/SINGLE_POST_FILE_STANDARD.md`
9. `docs/operations/TOPIC_SELECTION_SCORECARD.md`
10. `outputs/reports/top10_analysis.md`
11. `outputs/reports/ranking_report.md`는 experimental 참고로만 읽는다. 특정 게시물 URL 기반 추적 구현 전까지 의사결정 근거로 사용하지 않는다.
12. 제목/도입부를 새로 쓰는 작업이면 `docs/strategy/HOOKING_FORMULA.md`를 읽는다.
13. 필요 시 `docs/strategy/BRAND_CONTEXT.md`, `docs/strategy/SEO_KEYWORD_RESEARCH.md`

읽은 뒤 반드시 아래 4가지를 먼저 판단한다.

| 판단 항목 | 기준 |
| --- | --- |
| 신규 글 작성인지 | `CONTENT_PLAN.md`의 다음 미작성 슬롯이 명확한 경우 |
| 리라이팅인지 | 순위 하락, 클릭은 있으나 순위 약함, 브랜드 방어 필요가 있는 경우 |
| URL 등록인지 | `POSTING_REGISTRY.md`에 URL이 `-`로 남아 있고 사용자가 URL을 제공한 경우 |
| 분석/점검인지 | 사용자가 순위, 조회수, 유입어, 리라이팅 계획을 요청한 경우 |

## 2. 현재 운영 기준

2026-06-13부터 원고 상태, 발행 상태, 운영 상태를 분리한다. `CONTENT_PLAN.md`의 체크는 원고 작성 완료를 뜻하며, 실제 네이버 발행 완료는 `POSTING_REGISTRY.md`의 URL과 발행일로 판단한다.

| 구분 | 상태값 |
| --- | --- |
| 원고상태 | 미작성, 작성중, 원고완료, 검수완료, 발행대기 |
| 발행상태 | 미발행, URL대기, 발행완료, 성과확인대기, 성과확인완료 |
| 운영상태 | 일반, 보호글, 보강후보, 리라이팅후보, 리라이팅발행본 |

운영 기준은 네이버 통계의 유입경로, 상세 검색어, 게시글 TOP 20, 등록부 상태를 우선한다. 기존 순위 추적기는 특정 게시물 URL 순위가 아니라 문장군 블로그 계정 첫 등장 위치에 가까우므로 아래 기준의 자동 근거로 쓰지 않는다.

신규 글감 판단은 네이버 통계만으로 끝내지 않는다. `TOPIC_SELECTION_SCORECARD.md`에 따라 네이버 광고 API 시장 수요를 시장 지도처럼 먼저 확인하고, 블로그 실제 유입어와 게시글 TOP20 반응으로 실전성을 검증한 뒤 문장군 서비스 적합성, AppSheet 현장 연결성, 기존 글과의 중복 여부, 발행 안전성을 함께 본다.

| 구분 | 기준 | 처리 |
| --- | --- | --- |
| 보호 글 | TOP 5 또는 전환 의도 강한 TOP 10 | 본문 리라이팅 금지. 내부링크/URL 등록만 허용 |
| 보강 글 | 6~15위 또는 조회수는 있는데 핵심 키워드 순위가 약한 글 | 제목, 도입부, 소제목, 내부링크 중심 보강 |
| 재작성 후보 | 16위 이하, TOP 30 밖, 브랜드 방어 실패 | 새 각도 리라이팅 또는 방어형 신규 글 검토 |
| 신규 글 | CONTENT_PLAN 미작성 슬롯 | 기존 등록부와 소재 중복 확인 후 작성 |

## 5. 신규 글 작성 프로토콜

1. `CONTENT_PLAN.md`에서 다음 미작성 슬롯 확인
2. `POSTING_REGISTRY.md`에서 같은 키워드와 같은 소재가 이미 있는지 확인
3. `TOPIC_SELECTION_SCORECARD.md` 기준으로 네이버 광고 API 시장 수요와 네이버 통계 유입어, 게시글 TOP 20에서 관련 키워드 반응을 함께 확인
4. 강한 키워드는 직접 중복 작성하지 말고 롱테일/사례/비용/단점 각도로 분화
5. `docs/strategy/POSTING_EXCLUSION_RULES.md`에서 완전 제외, 단독 제외, 검색어 전환, 현장 변수 가능 여부를 확인
6. `docs/operations/APPSHEET_FIELD_STORY_WORKFLOW.md` 기준으로 현장 조건과 AppSheet 후매칭 카드를 설계
7. AppSheet 원본 후보가 이미 있으면 현장 이야기, 구조 변수, 사진 활용 포인트, `evidence_ref` 범위를 먼저 정리
8. AppSheet 원본 후보가 아직 없으면 작성은 멈추지 않는다. 실제 고객 사례처럼 쓰지 않고 일반 현장 조건형 정보글로 작성하되, 직원이 후매칭할 사진 방향은 본문 문장 안에 자연스럽게 드러나게 쓴다.
8-1. `## 실제 시공 현장에서는 조금 다릅니다` 단락은 AppSheet 현장 서사 슬롯으로 쓴다. 고객 고민, 현장 확인 요소, 판단 기준, 진행 방향, 시공 후 변화가 들어가야 하며 직원이 실제 지역·아파트명으로 첫 문장을 치환해도 자연스러워야 한다.
9. `top10_analysis.md`에서 제목 패턴 확인. 제목 후보는 5개를 만들고, 검색형/현장형/통념반박형/손실회피형/서비스형 중 최소 3가지 이상을 섞어 "3가지" 제목 반복을 피한다. `ranking_report.md`는 URL 기반 추적 구현 전까지 자동 판단에서 제외
10. 글 작성 후 발행 본문은 `posts/NNN_키워드.md` 단일 파일에 저장하되, 파일 상단에는 `## 제목 후보 5개` 섹션을 둔다
11. 최종 발행 전 `humanize-korean` 스킬 기준으로 문체 윤문 패스를 적용한다. 이 단계는 내용 생성이 아니라 AI 티 제거용이다.
12. 윤문 중에도 사실, 수치, 제품명, 지역명, 현장 조건, 직접 인용, CTA는 바꾸지 않는다. 문장군 현장 서사와 AppSheet 현장 슬롯이 약해지면 윤문을 롤백한다.
13. 별도 제작노트는 만들지 않는다. 사진 큐, AppSheet 확인, 제작자 메모, 운영 메모는 원고 안에도 남기지 않는다
14. 실제 사례, 현장 사진, 직접 인용, 강한 성능 주장을 쓰면 `outputs/publish_control/NNN_키워드/EVIDENCE.json`에 근거 범위를 기록
15. `npm run validate:posts`로 자동 검수하고 결과를 `outputs/checks/`에 저장
16. `CONTENT_PLAN.md` 상태는 원고완료 기준으로 업데이트
17. `POSTING_REGISTRY.md`에 파일, 제목, 소재 요약과 상태를 등록
18. 발행 전 `docs/operations/PREPUBLISH_CHECKLIST.md`를 확인
19. 발행 후 사용자가 URL을 주면 URL과 발행일 등록

### AppSheet 현장 매칭 기준

신규 글은 검색어 대응형이어도 최소 1개의 현장 축을 가져야 한다.

| 글 유형 | AppSheet에서 찾을 것 |
| --- | --- |
| 비용형 | 견적 범위가 달라진 구조 변수, 추가 확인 항목, 시공 범위 차이 |
| 구조형 | 현관 폭, 신발장, 스위치, 벽면, 가벽, 몰딩, 바닥 수평 |
| 제품형 | 제품 선택이 바뀐 이유, 유리/프레임/구동 방식 조합 |
| 거주중 시공형 | 작업 시간, 생활 동선, 소음·분진, 집에 있어야 하는지 |
| 유지보수형 | 손상 원인, 교체 범위, 사진으로 설명 가능한 전후 차이 |

실제 고객 사례처럼 쓰는 순간 `EVIDENCE.json`이 필요하다. 근거가 없는 경우에는 "실제 고객님 사례"가 아니라 "현장에서 자주 확인되는 조건"으로 표현한다.

098번 이후 통계 기반 신규 글은 예전 019/050번대 스타일을 기준으로 현장 상담을 재현하듯 쓴다. 단순 체크리스트로 짧게 끝내지 않고, 공백 제외 본문 1,500~2,500자를 합격 범위로 본다. 길이는 보조 기준이며, 글 방향은 검색 의도·브랜드 컨텍스트·현장 서사가 먼저다.

현장 서사 슬롯 필수 구조:

```text
이번/비슷한 현장은 [지역·아파트·구조로 치환 가능한 상황]이었다
→ 고객은 [처음 고민] 때문에 망설였다
→ 현장에서 [폭·신발장·스위치·레일·문짝 위치]를 확인했다
→ 문장군은 [판단 기준]으로 [진행 방향]을 잡았다
→ 시공 후 [사진으로 보이는 변화]가 확인됐다
```

대괄호는 원고에 남기지 않는다. 실제 원고는 모두 자연문으로 작성한다.

### 제품/시공 범위 보정 기준

2026-06-15 사장님 오더 기준으로 아래 표현은 신규 원고에서 반드시 지킨다.

| 소재 | 작성 기준 |

| 중문파티션 | 이번 신규 작성 후보에서 제외한다. |
| 비대칭양개형중문 | 문장군 미취급 제품이므로 작성 대상에서 제외한다. |
| 살면서리모델링 | 종합 리모델링으로 쓰지 않는다. `살면서 방문교체`, `거주중 중문시공`으로 문장군 실제 시공 범위에 맞춰 좁힌다. |
| 접이식중문 | 소비자 검색어로만 활용하고, 본문은 문장군 제품명인 `스윙` 구조로 안내한다. |
| 9mm문선 | 검색어 대응·비교 설명만 가능하다. 문장군 시공 가능 서비스처럼 쓰지 않는다. 문장군 기준은 `12mm슬림문선 세트`다. |
| 신발장리폼 | 제외 제품이 아니다. 단독 신발장 설치 글로 쓰지 않고, 중문 설치 시 신발장 돌출·측면 가공·스위치 간섭·추가금 변수로 쓴다. |

상세 기준은 `docs/strategy/POSTING_EXCLUSION_RULES.md`를 우선한다.

### 단일 발행 MD 규칙

`posts/` 파일은 네이버 블로그에 그대로 붙여넣을 발행 본문만 담는다.

문체 윤문을 위해 `humanize-korean` 스킬을 사용하더라도 최종 산출물은 `posts/NNN_키워드.md` 하나다. `_workspace/.../final.md`나 윤문 요약 파일은 임시 참고물이며, 운영 산출물로 등록하지 않는다.

`posts/`에 남기는 것:
- 최종 제목
- 본문
- 관련 글 링크
- 태그란 입력용 해시태그

별도 제작노트로 만들지 않는 것:
- 검색 의도 분석
- 썸네일 카피
- 품질 채점 결과
- 예상 성능
- 이미지 삽입 지시
- AppSheet 후매칭 조건
- 사진 삽입 지도
- 발행 전 작업 메모

위 항목은 원고 품질을 위해 머릿속에서 판단하되, 산출물로 남기지 않는다. 사진 매칭 방향은 본문 문장 구조 안에 자연스럽게 녹인다.

제목 후보 5개는 별도 제작노트가 아니라 `posts/NNN_키워드.md` 상단의 `## 제목 후보 5개` 섹션에만 남긴다. `posts/` 파일이 옛 제작노트 형식인 `# 제목 후보`로 시작하면 발행 불가로 본다.

중요: 019번처럼 한 파일에 제목 후보, 검색 의도, 품질 채점, 예상 성능, 이미지 지시가 같이 있던 방식은 과거 형식이다. 2026-06-24 이후에는 파일을 쪼개지 않고, `posts/`에는 네이버에 그대로 붙여넣을 최종 발행 본문만 남긴다. 글맛과 구조는 019/050번대 성공 글을 참고하되, 내부 메모와 사진 지시문은 남기지 않는다.

## 6. 리라이팅 프로토콜

리라이팅은 무조건 아래 순서를 따른다.

1. 네이버 통계 유입어, 게시글 TOP 20, 등록부의 URL/소재 상태 확인
2. `ranking_report.md`의 순위와 하락 폭은 experimental 참고로만 보고, 특정 글 리라이팅 근거로 단독 사용하지 않는다
3. 기존 글의 타겟 키워드, 제목, 도입부, 소제목, 내부링크를 점검
4. 기존 강점을 훼손하지 않는 보강안부터 제안
5. 제목 변경이 필요한 경우 기존 URL과 발행 이력 리스크를 명시
6. 리라이팅 원고는 반드시 새 번호의 새 파일로 저장한다
7. 기존 원본 파일은 절대 덮어쓰지 않는다
8. `POSTING_REGISTRY.md`에는 새 행으로 등록하고, 소재 요약에 원본 참조 번호를 남긴다

### 리라이팅 파일명 규칙

리라이팅은 기존 글 수정본이 아니라 새 발행본이다. 파일명은 아래 규칙을 강제한다.

```text
posts/NNN_키워드_리라이팅.md
```

예시:

```text
posts/068_문장군중문_리라이팅.md
posts/069_3연동중문_리라이팅.md
posts/070_아파트현관중문_리라이팅.md
```

금지:

```text
posts/002_3연동중문.md        # 기존 원본과 충돌
posts/002_3연동중문_RE.md     # 기존 번호 재사용
posts/3연동중문_리라이팅.md   # 번호 없음
```

`POSTING_REGISTRY.md` 등록 형식:

```text
| 069 | 069_3연동중문_리라이팅.md | 리라이팅(H2) | 3연동중문 | 새 제목 | - | - | 002_3연동중문.md의 순위 회복용 새 발행본. ... |
```

리라이팅 판단 기준:

| 상황 | 권장 작업 |
| --- | --- |
| 1~5위 | 리라이팅 금지 |
| 6~10위 | 내부링크, CTA, 이미지 설명 보강 |
| 11~20위 | 제목 후보와 도입부 보강 검토 |
| 21위 이하 | 구조 보강 또는 새 각도 리라이팅 |
| TOP 밖 | 방어형 신규 글 또는 전면 리라이팅 후보 |

## 7. 제목 운영 규칙

2026-06-11 TOP10 분석 기준을 따른다.

- 제목 후보 5개 필수: `posts/` 파일 상단에 `## 제목 후보 5개` 섹션을 두고 후보 5개를 함께 남긴다. 실제 발행 제목은 그 아래 첫 `# 제목`으로 둔다.
- 숫자 포함 우선: 상위 120개 글 중 41%. 다만 숫자형은 우선 검토일 뿐 모든 글 제목을 "3가지"로 고정하지 않는다.
- 질문형 제목 금지: 상위 글 질문형 0%
- 평균 제목 길이: 31자 안팎
- 키워드 앞배치
- `중문` 단독 제목 금지. `아파트 중문`, `현관 중문`, `3연동중문`처럼 복합 키워드 사용
- 비용/가격/설치 의도가 강할 때는 제목에 `비용`, `가격`, `설치`, `추가금`, `체크`를 적극 사용
- 제목과 도입부는 `docs/strategy/HOOKING_FORMULA.md`의 후킹 공식을 적용한다.
- 문장군식 후킹은 자극보다 `검색자의 오해 반박`, `현장 변수`, `손실 회피`, `타깃 호명`을 우선한다.
- 근거 없는 수치 결과, 실제 사례처럼 보이는 표현, 직접 인용문, 강한 보장 표현은 후킹에 쓰지 않는다.

제목 후보 5개 구성 예시:

1. 검색형: 핵심 키워드와 검색 의도를 가장 선명하게 담은 제목
2. 현장형: 실제 현장 변수나 고객 고민을 앞세운 제목
3. 통념반박형: 고객이 흔히 오해하는 지점을 바로잡는 제목
4. 손실회피형: 추가금, 실패, 후회 방지 관점을 담은 제목
5. 서비스형: 무료 방문 실측, 일정, 상담 경험 등 문장군 서비스 강점을 담은 제목

## 8. 포스팅 후 루프

발행 후 사용자가 URL을 주면 다음 순서로 처리한다.

1. `POSTING_REGISTRY.md` URL 등록
2. 제목이 깨져 있으면 사용자가 준 실제 제목으로 교정
3. 네이버 통계 유입어와 게시글 TOP 20을 일일 리포트에 기록
4. `npm run analyze`
5. 필요할 때만 `npm run track`을 experimental 보조 지표로 실행
6. 유입어와 게시글 TOP 20을 기준으로 이 플레이북의 큐를 조정할 필요가 있는지 판단
