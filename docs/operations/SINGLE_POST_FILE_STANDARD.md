# 단일 발행 MD 운영 기준

> 최종 업데이트: 2026-06-26

## 1. 결론

문장군 블로그 원고는 `posts/NNN_키워드.md` 한 개만 만든다.

단일 발행 MD 원칙은 로컬 작업 기준이다. `posts/` 원고 본문은 GitHub에 커밋하지 않고, 공개 저장소에는 운영 문서와 registry/queue 같은 운영 OS만 남긴다.

`outputs/drafts/NNN_키워드_note.md` 같은 제작노트는 만들지 않는다. 사장님과 직원이 보지 않는 파일은 운영상 없는 파일과 같고, 파일을 쪼개면 사진 매칭과 검수 기준이 흩어진다.

## 2. 단일 파일 원칙

- `posts/`의 MD 파일은 네이버에 그대로 붙여넣을 수 있는 발행 본문이어야 한다.
- 발행 본문 안에 `[사진:]`, `[AppSheet 확인]`, `[제작자 메모]`, `[제작노트]` 같은 내부 지시문을 넣지 않는다.
- `## 운영 메모` 같은 숨은 내부 메모 섹션도 만들지 않는다.
- 사진 매칭 방향은 별도 지시가 아니라 본문 문장 구조 안에 자연스럽게 녹인다.

## 3. 사진 큐를 본문에 녹이는 방식

나쁜 방식:

```markdown
[사진: 신발장 문 열린 상태 삽입]
신발장 문이 열리는 방향을 먼저 봐야 합니다.
```

좋은 방식:

```markdown
신발장 간섭은 사진 한 장만으로 판단하기 어렵습니다.

현관 전체가 넓어 보여도 신발장 문을 실제로 열어보면 중문 프레임이 들어갈 선과 겹치는 경우가 있습니다. 그래서 상담 전 사진을 찍을 때는 현관 정면 사진뿐 아니라 신발장 문을 연 상태도 함께 확인하는 것이 좋습니다.
```

이렇게 쓰면 직원은 자연스럽게 `현관 전체`, `신발장 문 열린 상태`, `프레임이 들어갈 선`이 보이는 사진을 찾게 된다. 내부 지시문은 없지만 사진 매칭 방향은 살아 있다.

## 4. 권장 본문 구조

```markdown
# 제목

도입부: 고객 고민과 검색 의도

## 1. 첫 번째 판단 기준
사진이 붙기 좋은 현장 장면을 본문 문장으로 설명

## 2. 두 번째 판단 기준
클로즈업해야 할 구조 요소를 자연스럽게 설명

## 3. 세 번째 판단 기준
Before/After가 붙기 좋은 문장 배치

## 실제 시공 현장에서는 조금 다릅니다
AppSheet 실제 현장으로 치환·확장 가능한 현장형 이야기

## 상담 전 이렇게 준비하면 좋습니다
고객이 찍어오면 좋은 사진과 확인할 조건을 본문으로 안내

## 관련 글

## 해시태그
```

## 5. AppSheet 현장 표현 기준

실제 AppSheet 근거가 있으면 지역, 제품, 구조, 사례 유형이 맞는 범위에서 현장 이야기를 쓴다.

`실제 시공 현장` 단락은 반드시 직원이 AppSheet 현장으로 바꿔 끼울 수 있는 자연문 구조여야 한다.

좋은 구조:

```markdown
실제로 좁은 현관 폭 때문에 중문 설치를 망설이셨던 현장도 있었습니다.

고객님은 처음에 3연동중문을 생각하고 계셨지만, 현관 가로폭이 좁아 보여서 설치 후 신발 신는 공간이 더 답답해질까 걱정하셨습니다.

현장에서 가로폭과 신발장 끝선, 문짝이 모이는 위치를 함께 확인해보니 단순히 넓고 좁음만 볼 문제는 아니었습니다.

이 현장은 실제 동선을 확인한 뒤 3연동중문 진행 방향을 잡을 수 있었습니다.
```

이 문단은 실제 AppSheet 현장을 찾으면 첫 문장을 `노원구 상계주공아파트 현장도 비슷했습니다`처럼 바꾸고, 현장 사진을 붙여도 자연스럽다.

근거가 아직 확정되지 않았으면 실제 고객 사례처럼 단정하지 않는다. 이때는 아래처럼 일반 현장 조건형으로 쓴다.

```markdown
실제로 현장 상담에서는 비슷한 고민이 자주 나옵니다.

처음에는 원슬라이딩 중문만 생각하고 계셨지만, 현장에서 신발장 문 열림과 프레임이 들어갈 위치를 함께 확인해보면 단순히 제품만 고를 문제가 아닌 경우가 있습니다.
```

근거 없는 세부 수치, 특정 지역, 특정 고객 발언, 실패 사례는 쓰지 않는다.

## 6. 검수 기준

`scripts/validate_post.js`는 아래 항목을 FAIL로 처리한다.

- `## 운영 메모` 섹션
- `[사진:]`, `[이미지:]` 형태의 사진 큐
- `[AppSheet 확인]`, `[앱시트 확인]`
- `[제작자 메모]`, `[제작노트]`, `[메모]`

문장군 블로그의 기준은 단순하다.

**한 파일, 발행 본문만, 사진 방향은 문장 안에.**

## 7. 문체 윤문 스킬 사용 기준

`humanize-korean` 스킬은 최종 문체 윤문용으로만 사용한다.

사용 목적:

- 번역투 줄이기
- 기계적인 병렬 문장 줄이기
- 접속사 남발 줄이기
- 문장 길이와 종결 리듬 다듬기
- "결론적으로", "이처럼", "이를 통해" 같은 AI식 연결 표현 줄이기

금지:

- 새로운 사례 추가
- 제품명, 지역명, 수치, 날짜 변경
- 실제 고객 발언처럼 보이는 문장 추가
- AppSheet 근거 없는 현장명 추가
- CTA 방향 변경
- 문장군 미취급 제품을 자연스럽게 보이도록 완화

`humanize-korean`이 `_workspace/{run_id}/final.md`를 만들 수 있지만, 그 파일은 임시 윤문 결과다. 문장군 블로그의 최종 원고는 항상 `posts/NNN_키워드.md` 하나여야 한다.

윤문 후에도 아래 구조는 유지되어야 한다.

```text
문제 제기
→ 공감
→ 판단 기준 3가지
→ 실제 시공 현장에서는 조금 다릅니다
→ 상담 전 준비
→ 관련 글
→ 해시태그
```

문체가 자연스러워졌더라도 현장 서사 슬롯이 약해지면 실패다.

## 제목 후보 보고 기준

단일 발행 MD 원칙은 유지한다. `posts/NNN_키워드.md` 파일 하나 안에 제목 후보 5개와 발행 본문을 함께 둔다.

파일 상단에는 `## 제목 후보 5개` 섹션을 두고, 그 아래 첫 `# 제목`을 실제 발행 제목으로 둔다.

제목 후보 5개는 같은 문장틀을 반복하지 않는다. 특히 모든 제목을 `3가지` 구조로만 만들지 않는다.

제목 후보는 반드시 핵심 키워드를 기반으로 만든다. 키워드를 버리고 감성 문장만 만들면 검색 유입이 약해진다. 반대로 키워드명만 붙이면 고객이 클릭할 이유가 약하다. 제목은 `키워드 + 고객 상황/불안`의 결합이어야 한다.

기본 구성:

1. 검색형 제목
2. 고객불안형 제목
3. 현장형 제목
4. 통념반박형 제목
5. 손실회피형 제목
6. 서비스/브랜드형 제목

제목 후보를 만들기 전 아래 3줄을 머릿속에서 먼저 확정한다.

```text
핵심 키워드:
고객 상황:
고객 불안/의심 문장:
```

최종 제목은 사용자가 선택하거나, 총괄이 검색 의도와 글맛을 함께 보고 1개를 추천한다. 제목 후보 외에 검색 의도, 품질 채점, 이미지 지시, AppSheet 확인 같은 내부 메모는 남기지 않는다.
